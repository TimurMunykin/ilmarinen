# Ilmarinen Platform Design

Personal app generator for non-technical users. Users describe a problem in natural language, the platform generates, builds, and deploys a web application.

## Context

The project builds on an existing stack proven in two similar applications (auf — dog care tracker, bloodyssey — blood test manager). Both share: NestJS + React + Prisma + PostgreSQL + Docker + Caddy, deployed behind muntim-gateway on a single VPS.

Ilmarinen automates the pattern of creating these apps — providing a shared boilerplate for infrastructure and using AI to generate business logic.

## Architecture

Three components:

### 1. Ilmarinen Platform

Web application (NestJS + React + Prisma + PostgreSQL + Docker) where users create and manage their apps.

Contains:
- **Authentication** — Google OAuth + JWT (same pattern as auf/bloodyssey)
- **Dashboard** — list of user's apps with status, links, and "edit" / "create new" actions
- **Chat interface** — hybrid UX: user describes problem in free text, AI asks clarifying questions, proposes a solution, user approves or requests changes
- **Admin panel** — manage AI access requests (approve/reject)
- **AI integration** — OpenAI API for conversation and code generation

### 2. App Template

Git repository containing the boilerplate for every generated app. Includes:

**Templated (static, same for every app):**
- `docker-compose.yml`, `docker-compose.prod.yml`
- `Dockerfile` for api and web
- `Caddyfile`
- `bin/deploy.sh`, `scripts/dev-start.sh`
- `package.json` (root workspace with Bun)
- `apps/api/src/main.ts`, `app.module.ts` (NestJS skeleton)
- `apps/web/src/main.tsx`, root route, theme-provider
- Auth module (Google OAuth + JWT via shared Ilmarinen auth)
- Base UI kit (shadcn/ui components)
- Notifications module (Telegram integration)
- Settings page (Telegram connection)

**Generated by AI (unique per app):**
- `apps/api/prisma/schema.prisma` — data models
- `apps/api/src/modules/*` — NestJS business modules (controllers, services)
- `apps/web/src/routes/*` — React pages
- `apps/web/src/components/*` — custom components
- `apps/web/src/lib/i18n/` — translations

### 3. App Engine

Service within the platform that orchestrates app creation and deployment.

Pipeline:

1. **Specification** — AI converts chat dialogue into structured JSON spec:
   - App name and subdomain
   - Data models with fields and relationships
   - Screens (list, detail, form types)
   - Notification rules (trigger conditions, channel, message templates)

2. **Scaffolding** — clone app-template, substitute name/subdomain in configs, CREATE DATABASE in shared PostgreSQL

3. **Code generation (AI)** — AI receives spec + template code, generates:
   - Prisma schema
   - NestJS modules (controllers, services, DTOs)
   - React pages and components
   - i18n translations

4. **Validation** — `bunx prisma validate` + `bunx tsc --noEmit`. On failure, AI receives error log and retries (up to 3 attempts)

5. **Build and deploy** — `docker build` for api and web, `docker compose up`, `prisma migrate deploy`, register subdomain route in Caddy

6. **Notification** — user receives link in chat: `<subdomain>.apps.muntim.ru`

Expected time: 2-5 minutes per app creation. User sees progress updates in chat.

## User Flow

### Creating an app

1. User visits `ilmarinen.muntim.ru`, logs in via Google
2. Empty dashboard → "Create app" button
3. Chat opens. User describes problem: "I need to remember when to vaccinate my dog and get reminders a week before"
4. AI asks clarifying questions: "How many dogs? What other procedures? How often to remind?"
5. AI forms spec, shows text preview of proposed screens and features
6. User approves or requests changes
7. App Engine generates, builds, deploys
8. User gets link: `my-pet.apps.muntim.ru`

### Editing an app

1. User visits Ilmarinen, sees app list
2. Clicks "Edit" on an app
3. Chat opens with history. User writes: "Add cat vaccination tracking too"
4. AI understands existing app context, proposes changes
5. Rebuild and redeploy

### AI access

1. New user creating first app sees: "Enter OpenAI API key" + "Request free access" button
2. Own key — stored encrypted (AES-256), used for their requests
3. Free access request — admin receives notification, approves in admin panel
4. After approval — user creates apps using platform's key

## Data Model (Platform)

- **User** — googleId, email, name, locale
- **App** — name, subdomain, status (creating/running/stopped/error), spec (JSON), generatedAt, deployedAt, userId
- **AppVersion** — appId, version, spec, gitCommitHash
- **ChatSession** — appId (nullable — null while app not yet created), userId
- **ChatMessage** — sessionId, role (user/assistant), content
- **AiAccessRequest** — userId, status (pending/approved/rejected)
- **AiKey** — userId (nullable — null for platform shared keys), encryptedKey, provider (openai)

Each generated app has its own separate database with its own Prisma schema generated by AI.

## Telegram Notifications

Single Telegram bot for the entire platform (`@ilmarinen_bot`).

**User connection:**
1. Generated app has "Settings" page (part of template)
2. "Connect Telegram notifications" button
3. Opens `t.me/ilmarinen_bot?start=<token>`
4. User presses /start — bot links chatId to user
5. Done

**Sending notifications:**
- Template includes `notifications` module
- AI generates notification rules during app creation
- Cron job inside each app's API checks schedule, sends via Telegram Bot API
- Single bot, but each app knows its user's chatId
- Message format includes app name for clarity when user has multiple apps

## Infrastructure

```
VPS
├── muntim-gateway (Caddy)          — existing, entry point
│   ├── ilmarinen.muntim.ru         → ilmarinen-web
│   ├── *.apps.muntim.ru            → wildcard route → app containers
│   ├── auf.muntim.ru               → auf (existing)
│   └── bloodyssey.muntim.ru        → bloodyssey (existing)
│
├── ilmarinen (platform)
│   ├── ilmarinen-api                — NestJS API
│   ├── ilmarinen-web                — React frontend
│   ├── ilmarinen-db                 — PostgreSQL (platform DB + all app DBs)
│   └── ilmarinen-telegram-bot       — long polling
│
└── generated apps
    ├── my-pet-api                   — container
    ├── my-pet-web                   — container (Caddy)
    ├── budget-tracker-api
    ├── budget-tracker-web
    └── ...
```

**Key decisions:**
- Single PostgreSQL server, separate database per app (saves RAM, data isolated)
- Wildcard DNS: `*.apps.muntim.ru → VPS IP` (configured once)
- Wildcard Caddy route in muntim-gateway (auto-updated on new app deploy)
- All containers on shared Docker network `muntim_gateway`
- Telegram bot: long polling (simpler than webhooks for start)

**Capacity:** 10-50 users, ~50-100 apps. Each app ~2 containers, ~100-200MB RAM. Single VPS sufficient for start.

## Security and Isolation

- **Database isolation** — separate database per app (not schema, full database). One app cannot read another's data.
- **Container isolation** — Docker containers isolated. Shared network only for routing.
- **Per-app JWT_SECRET** — generated at creation time.
- **AI key encryption** — AES-256, decrypted only at request time.
- **Code generation boundaries** — AI generates only within designated directories (modules, routes, components). Cannot modify Docker, deploy scripts, auth.
- **Validation gate** — TypeScript + Prisma validation before deploy catches syntax errors.
- **Authentication** — shared Google OAuth through Ilmarinen as auth provider. Single login works across platform and user's apps.

## MVP Scope

### In scope
- Ilmarinen platform (auth, dashboard, chat, AI access admin)
- App Template (boilerplate with auth, UI kit, Telegram module, Docker)
- App Engine (spec → scaffolding → AI code generation → validation → build → deploy)
- Single Telegram bot for notifications
- Wildcard DNS + Caddy routing
- Shared Google OAuth for platform and apps
- OpenAI as sole LLM provider
- RU/EN localization

### Out of scope (future)
- Custom domains (only `*.apps.muntim.ru`)
- Other LLM providers (Anthropic, open models)
- Other notification channels (email, push, SMS)
- Visual app editor (chat only)
- App export (download code)
- App monitoring/analytics
- Version rollback UI (history stored, rollback UI later)
- Multi-server scaling
- Paid subscription / billing
